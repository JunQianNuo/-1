## 2. 当前网格评价器的内存估算

当前 `coverage_counts` 已改为按时间批处理：

```python
batch_size = max(1, min(L, 10))
for start in range(0, L, batch_size):
    dots_batch = np.einsum("sbD,kD->sbk", sat_batch, ground)
```

因此它没有一次性生成旧式的

$$
(S,K,L)
$$

三维点积矩阵。若不分批，对 $S=1600$、$K=12375$、$L=2874$ 的细网格而言，仅 float64 点积矩阵就约为

$$
1600\times12375\times2874\times8
\approx455\,\mathrm{GB},
$$

明显超过 16GB。

当前分批后，主要数组包括：

- `sat_all`：形状 $(S,L,3)$；
- `counts`：形状 $(K,L)$；
- `dots_batch`：形状 $(S,B,K)$，其中 $B=10$；
- `covered`：形状 $(S,B,K)$ 的布尔矩阵。

在 $S=1600$ 时，估算如下：

| 网格与时间设置                              |   $K$ |  $L$ | `sat_all` | `counts` | `dots_batch` |  估计核心峰值 |
| :----------------------------------- | ----: | ---: | --------: | -------: | -----------: | ------: |
| $3^\circ$，$180\,\mathrm{s}$，23.934h  |   396 |  480 |   0.02GiB |    约0GiB |      0.05GiB | 0.07GiB |
| $1^\circ$，$60\,\mathrm{s}$，23.934h   |  3150 | 1438 |   0.05GiB |  0.01GiB |      0.38GiB | 0.48GiB |
| $0.5^\circ$，$30\,\mathrm{s}$，23.934h | 12375 | 2874 |   0.10GiB |  0.07GiB |      1.48GiB | 1.83GiB |
|                                      |       |      |           |          |              |         |

所以，**单个候选的内存尚可控，但单个候选的计算时间很高**。在 16GB 内存机器上，不能通过增大 `batch_size` 来换速度，否则很容易触发内存峰值过高。

## 3. 当前代码的主要风险点

### 3.1 `run_q2_scan.py` 默认无候选上限

该脚本默认：

```text
single-start = 40
single-stop = 150
max-candidates = 0
duration-hours = 23.934
```

其中 `max-candidates=0` 会被解释为“不设上限”。按当前候选生成方式估算：

| 搜索范围 | 候选总数 |
|:--|--:|
| $S=40\sim80$ | 2,055,508 |
| $S=40\sim100$ | 3,557,092 |
| $S=40\sim150$ | 8,735,328 |

这些候选不仅会导致运行时间极长，还会将所有候选记录保存在 `search.records` 中，进一步带来内存压力。

> [!danger] 不建议运行
> 不建议直接运行 `python run_q2_scan.py` 的默认参数。该脚本属于前一阶段枚举法尝试，不适合 16GB 内存和当前问题规模。

### 3.2 `run_q2_sweep.py` 会持续累积 `all_records`

脚本中存在：

```python
all_records: list[dict] = []
...
all_records.append(q2.evaluation_record(result))
```

如果枚举范围扩大，`all_records` 会持续增长。虽然每条记录不大，但百万级候选会带来显著内存占用。

此外，脚本后续硬编码了：

```python
lat_f, lon_f = q2.make_latlon_grid(step_deg=0.5)
times_f = q2.make_time_grid(duration_s=24 * 3600, step_s=30.0)
```

这一步对单个候选的内存可控，但耗时很高，不适合作为大规模循环内部步骤。

### 3.3 `search_single_coverage` 和 `search_double_coverage` 保存完整 records

当前搜索函数每评价一个候选都会执行：

```python
records.append(evaluation_record(result))
```

这对 smoke run 或小规模粗搜索没问题，但不适合百万级候选。大规模搜索时应改为：

1. 只保留 top-k 候选；
2. 或边评价边流式写入 CSV；
3. 或只记录每个 $S$ 的最佳候选。

### 3.4 `satellite_unit_vectors` 仍一次性生成全部时间的卫星位置

当前函数返回：

$$
(S,L,3)
$$

数组。对 $S=1600$、$L=2874$ 而言约为 0.10GiB，可接受；但若未来采用更细时间步或更大星座，该部分也应改为按时间块生成，而不是全时段一次性生成。

## 4. 16GB 内存下的安全策略

在 16GB 内存约束下，建议遵守以下硬规则：

| 项目 | 建议 |
|:--|:--|
| 单次 `dots_batch` | 不超过约 2GiB |
| 网格法用途 | 只用于少量候选复核 |
| 大规模搜索 | 使用临界点法，不使用全网格法 |
| 候选记录 | 不保存全量 records，只保存 top-k 或流式写出 |
| 默认枚举脚本 | 应停止使用或加安全上限 |
| 最终复核 | 细网格只对 1 到 5 个候选执行 |

若继续保留网格法，建议加入内存估算：

$$
M_{\mathrm{dots}}
=
8SBK,
$$

其中 $B$ 是时间批大小。若限制 `dots_batch` 不超过 $2\,\mathrm{GiB}$，则

$$
B
\le
\frac{2\cdot1024^3}{8SK}.
$$

当 $S=1600$、$K=12375$ 时，

$$
B\le13.55.
$$

所以当前 `batch_size=10` 是合理的；不应随意增大到 50 或 100。


并按以下顺序实现：

1. `neighbor_pairs_by_dot`：只找距离不超过 $2\theta$ 的卫星对；

## 8. 搜索方向修正：从可行上界向下搜索

由于已有证据显示 $S=1600$ 可达到单重覆盖要求，因此正式搜索不宜从 $S=40$ 或 $S=900$ 盲目向上扫。更合理的策略是：

```text
已知可行上界 S=1600
→ 从 1600 向下搜索
→ 找到仍可行的较小 S
→ 在临界转折区间内加密候选与时间步长
→ 对少量候选做细网格复核
```

为支持该策略，`run_q2_fast_search.py` 已加入：

1. `--search-order desc`：按 $S_{\max},S_{\max}-1,\ldots,S_{\min}$ 降序搜索；
2. `--stop-if-min-count-below 1`：默认早停机制，候选一旦在任一快速评价时间片上出现 $\min c<1$，立即终止该候选后续时间片计算；
3. 快速结果中新增 `fast_stopped_early` 与 `fast_evaluated_time_steps` 字段，用于判断候选是否被提前淘汰；
4. 流式落盘文件 `q2_fast_search_stream_fast_records.csv` 和 `q2_fast_search_stream_verified_records.csv`，避免长跑中断后没有任何结果；
5. 按因子对均衡抽样，优先测试接近平衡的多轨道面构型，避免 `max_candidates_per_total` 截断时全部落在 `M=1,N=S`。

`S=1000` 单候选基准测试表明，单时刻快速评价约为 $0.31\sim0.33\,\mathrm{s}$，5 个时间片约为 $1.48\,\mathrm{s}$。因此搜索瓶颈不是内存，而是：

$$
\text{候选数}\times \text{时间片数}\times \text{单时间片临界点评价耗时}.
$$

推荐下一条粗搜索命令为：

```bash
python run_q2_fast_search.py --start-total 1200 --stop-total 1600 --search-order desc --inclinations 50,53,55,58,60 --phase-resolution 30 --max-candidates-per-total 200 --keep-top-fast 50 --verify-top 5 --duration-hours 6 --fast-time-step 900 --verify-time-step 900 --verify-grid-step 6 --output-dir results/fast_search_S1600_down_coarse
```

这一步仍是粗搜索，若发现可行候选集中在某一 $S$ 区间，应再缩小 $S$ 区间并加密倾角、相位和时间步长。

实际试跑发现，旧候选截断方式在 `S=1600`、`max_candidates_per_total=20` 时全部取到 `M=1,N=1600`，不具备代表性。修正为均衡因子对抽样后，`S=1600` 小批候选覆盖了 `40\times40`、`32\times50`、`50\times32`、`64\times25` 等构型。一次 20 候选、6 小时粗时间片的小批运行耗时约 61 秒，当前最佳网格复核候选为：

```text
M=40, N=40, i=50°
C1≈0.9984, C2≈0.9830, c_min=0
```

这说明粗网格下已接近单重覆盖，但仍存在空窗或边缘未覆盖点，后续应围绕多轨道面构型和相位参数继续搜索，而不是扩大 `M=1` 类构型。
